## ЁЯЪА Problem Highlights

### Day 1 тАФ Rotate List
**The trick:** Don't shift nodes one by one. Connect the tail to the head to make a circle, walk to the new tail at index `length тИТ k тИТ 1`, then break the circle. Two passes, no extra memory.

### Day 2 тАФ Rotating the Box
**The trick:** The 90┬░ clockwise rotation maps "rightward" in the original to "downward" in the result. Apply gravity first (stones slide right per row using a two-pointer), then apply the standard rotation formula `result[j][m-1-i] = box[i][j]`. Each obstacle resets the gravity pointer independently.

### Day 3 тАФ Jump Game IX
**The non-obvious bug fix:** The standard PGE + NSE spanning tree fails for arrays like `[30, 21, 5, 35, 24]`. When components merge, a future small element must compare against the **component's max value**, not the individual element's value. Maintain a stack of `(component_root, component_max)` pairs тАФ pop and union whenever `component_max > nums[i]`. Single pass, no second scan needed.

### Day 4 тАФ Minimum Jumps via Prime Teleportation
**The trick:** Naive BFS is O(n┬▓) тАФ prime 2 alone could fan out to half the array. Fix: group all indices by their prime factors into `prime_buckets[p]` upfront. First time any index with `nums[i] = p` is dequeued, drain and **clear** its bucket. Future activations find it empty тЖТ zero re-work. Uses a smallest-prime-factor (SPF) sieve for O(log V) factorization per number.

### Day 5 тАФ Roman to Integer
**The trick:** Iterating right-to-left turns the subtraction rule into a simple comparison. If the current numeral is smaller than the one to its right (e.g., `I` before `V`), subtract it; otherwise add it. No look-ahead needed тАФ a clean single pass with a fixed 7-entry hash map keeps it at O(n) time and O(1) space.

### Day 6 тАФ Cyclically Rotating a Grid
**The trick:** A 2D ring rotation is just a 1D list rotation in disguise. Traverse each layer clockwise into a flat list, shift it left by `k % layer_size` positions (`elements[k:] + elements[:k]`), then write values back in the same traversal order. Each layer is fully independent тАФ no other layer is affected.

### Day 7 тАФ Maximum Number of Jumps to Reach the Last Index
**The trick:** Unlike most jump games that ask for the minimum, this one asks for the maximum тАФ so we want as many small hops as possible. Greedy fails because an "easy" jump now can lead to a dead end. Bottom-up DP fills `dp[j] = max(dp[i] + 1)` for all valid predecessors `i`, where validity means `|nums[j] - nums[i]| тЙд target`. Mark unreachable indices as `-1` so they're never used as springboards.

### Day 8 тАФ Minimum Initial Energy to Finish Tasks
**The trick:** Sort tasks by **`minimum тИТ actual`** descending тАФ the "surplus" or wasted-energy buffer. Tasks with the largest surplus have the highest entry cost relative to their actual spend, so they must be done first while energy reserves are full. After sorting, simulate: top up only when forced. The sum of all top-ups is the minimum initial energy. A pairwise exchange argument proves the sort order is optimal.

### Day 9 тАФ Minimum Moves to Make Array Complementary
**The trick:** Iterate over the target sum `T тИИ [2, 2┬╖limit]` instead of over moves. For each pair `(a, b)`, the cost as a function of `T` has only **three zones** тАФ 0 moves at `T = a + b`, 1 move on `[1 + min(a,b), limit + max(a,b)]`, and 2 moves elsewhere. Apply these as range updates on a **difference array**, then prefix-sum and find the minimum. Beats the naive O(n┬╖limit) approach by a factor of n.

### Day 10 тАФ Check if Array is Good
**The trick:** The candidate `n` is forced тАФ it must equal `max(nums)`, because `base[n]` always has `n` as its largest element. Then two cheap checks suffice: `len(nums) == n + 1` (rejects the wrong-length cases instantly), and sorted `nums == [1, 2, тАж, nтИТ1, n, n]` (verifies all values and counts in one shot).

### Day 11 тАФ Find Minimum in Rotated Sorted Array
**The trick:** Compare `nums[mid]` to `nums[right]` (**not** `nums[left]`). If `nums[mid] > nums[right]`, the rotation pivot is strictly right of `mid` (`left = mid + 1`); otherwise the right half is sorted and the minimum is at `mid` or to its left (`right = mid`, **not** `mid тИТ 1`, so `mid` stays a candidate). One rule, no special-case for the no-rotation input.

### Day 12 тАФ Find Minimum in Rotated Sorted Array II
**The trick:** Duplicates introduce a third case: `nums[mid] == nums[right]` is genuinely ambiguous тАФ arrays like `[3,3,1,3]` and `[3,1,3,3,3]` give the same comparison with the minimum on opposite sides. **Fallback safely with `right -= 1`** тАФ if `nums[right]` were the unique minimum, `nums[mid]` would equal it, so `mid` still holds an equivalent copy. Worst case degrades to O(n) on adversarial inputs like `[3,3,3,3,1,3,3]`.

### Day 13 тАФ Jump Game III
**The trick:** Reframe as **graph reachability**: each index is a node with at most two outgoing edges (`i + arr[i]` and `i тИТ arr[i]`). BFS with a `visited` array runs in O(n). Check `arr[i] == 0` on dequeue тАФ that's the cleanest place, and it handles "started on a zero" without a special case.

### Day 14 тАФ Jump Game IV
**The O(n┬▓) trap:** Same-value teleports can connect any index to up to `n тИТ 1` neighbours тАФ `[7,7,7,тАж,7]` would be quadratic. **The fix:** group indices by value into buckets, and after the first time any index with value `v` is dequeued, **clear `buckets[v]`**. Every other index sharing that value has already been enqueued, so future visits would re-walk an irrelevant list. Each bucket is iterated at most once тЖТ total teleport work across the whole BFS is O(n).

### Day 15 тАФ Minimum Common Value
**The trick:** Both arrays are already sorted, so a **merge-style two-pointer scan** is the natural fit. Each step either finds the answer (`nums1[i] == nums2[j]` тАФ guaranteed minimum because both pointers only ever moved forward through smaller values) or advances whichever pointer has the smaller value (it can't be common with anything further ahead in the larger array). O(n + m) time, O(1) space тАФ strictly better than the hash-set approach.

### Day 16 тАФ Find the Prefix Common Array of Two Arrays
**The trick:** A **bitmask is a set in one integer**. Track `seen_A` and `seen_B` as bitmasks where bit `v` flips on once value `v` is seen. The common values at index `i` are exactly the bits set in **both** тАФ so `popcount(seen_A & seen_B)` gives `C[i]` in a single line. Permutations + `n тЙд 50` make this trivially efficient.

### Day 17 тАФ Find the Length of the Longest Common Prefix
**The trick:** Brute-forcing all `n┬╖m` pairs is too slow. Instead, **decouple** the arrays: insert every prefix of every number in `arr1` into a hash set using repeated `// 10` (integer division by 10 strips the last digit, generating prefixes longest-to-shortest for free). Then for each number in `arr2`, walk its own prefixes longest-first тАФ the first set hit is automatically the longest match, so `break` immediately. O((n + m) ┬╖ D) with `D тЙд 9` digits тАФ effectively linear.

### Day 18 тАФ Search in Rotated Sorted Array
**The trick:** A rotated sorted array has exactly **one break point**, so for any `mid`, **at least one of `[left..mid]` or `[mid..right]` is fully sorted**. Use `nums[left] <= nums[mid]` to identify the sorted half (the `<=` also handles `left == mid` and the no-rotation case). Then an O(1) range check on the sorted half decides direction: if `target` falls in its known range, recurse there; otherwise recurse the other half. One-pass O(log n).

### Day 19 тАФ Check if Array Is Sorted and Rotated
**The trick:** A non-decreasing array rotated by any amount has **at most one "drop"** (a position where the next element is smaller) тАФ the drop sits between the original max and the original min. Scan **circularly** using `(i + 1) % n`, which unifies the unrotated case (drop at the wrap) and the rotated case (drop internal) under one rule. Allow at most one drop тЖТ valid.

### Day 20 тАФ Jump Game V
**The trick:** Every jump goes from a strictly higher value to a strictly lower one, so the implicit graph is a **DAG** тАФ memoized DFS is safe. The killer micro-optimization is the **`break` on `arr[k] >= arr[i]`**: once any "in-between" index meets or exceeds `arr[i]`, it blocks both itself and every further index in that direction (it would violate the "lower-than-launch" rule for any longer jump). This reduces per-state work from O(d┬▓) to O(d), giving an overall O(n┬╖d). For deeper chains, a sort-by-height bottom-up variant avoids recursion entirely.

### Day 21 тАФ Jump Game VII
**The trick:** Reachability where each `'0'` index connects to every `'0'` in `[i + minJump, i + maxJump]`. Naive BFS rescans overlapping windows тЖТ O(n┬╖maxJump) TLE. The fix is a **sliding frontier pointer**: since BFS dequeues indices in non-decreasing order, track the largest index ever scanned and start the next window at `max(i + minJump, farthest + 1)`. Every index is examined at most once тЖТ O(n).

### Day 22 тАФ Count the Number of Special Characters I
**The trick:** We only need **presence**, not counts тАФ a letter is "special" if it appears in both cases at least once. Two hash sets (`lower_seen`, `upper_seen`) and a 26-bounded final tally do the job: count letters `c` where `c тИИ lower_seen` and `c.upper() тИИ upper_seen`. O(n) time, O(1) space.

### Day 23 тАФ Count the Number of Special Characters II
**The trick:** The added "every lowercase before the first uppercase" rule collapses to comparing **two indices per letter**: `last_lower[c] < first_upper[c.upper()]`. Why those two extremes? The latest lowercase is the hardest to beat, and the earliest uppercase is the lowest bar to clear тАФ if those pass, every actual pair does. Other index pairings (first/first, last/last) silently fail.

### Day 24 тАФ Longest Common Suffix Queries
**The trick:** "Longest common suffix" = "longest common prefix of the reverses." Build a **trie over reversed container strings**, and at every node pre-store the **best** container index passing through it (shortest length, then earliest). A query just walks `reversed(query)` down as far as it goes and returns the `best` at the deepest node тАФ the empty-suffix fallback drops out for free at the root. O(C + Q).

### Day 25 тАФ Minimum Element After Replacement With Digit Sum
**The trick:** The answer only needs the *minimum* of the digit-sum array, so stream each digit sum straight into `min()` тАФ no intermediate array. Extract digits with `% 10` / `// 10` (faster than string conversion). O(n) time, O(1) space. The digit sum of `nums[i] тЙд 10тБ┤` is bounded by 36.

### Day 26 тАФ Block Placement Queries
**The trick:** Obstacles carve the line into gaps; a block of size `sz` fits in `[0, x]` iff a usable gap тЙе `sz` lies inside. Store each gap at its **right endpoint** in a **max segment tree** (prefix-max query auto-excludes gaps past `x`), and add the trailing partial gap `x тИТ last`. Inserting an obstacle **splits** a gap into two point-updates тАФ one of them *decreases*, which is exactly why a Fenwick max won't work and a segment tree is required. O(n log M).

### Day 27 тАФ Destroying Asteroids
**The trick:** Mass only ever grows, so destroying the **smallest reachable asteroid first** is optimal тАФ sort ascending and sweep, absorbing as you go. Failure is genuine because every order absorbs the same cumulative mass: at the first asteroid you can't beat, you already hold the maximum possible mass. Provable via an exchange argument. (Mind 64-bit overflow outside Python тАФ mass reaches ~10┬╣тБ░.)

### Day 28 тАФ Minimum Cost of Buying Candies With Discount
**The trick:** The free candy in any triple is the cheapest of the three (`тЙд min(bought)`). To save the most, make freebies as *expensive* as legally allowed: **sort descending** and skip every index `тЙб 2 (mod 3)`. Sorting ascending would be both costlier *and* illegal (freeing a large candy violates the `тЙд min` rule).

### Day 29 тАФ Earliest Finish Time for Land and Water Rides I
**The trick:** For a fixed (land, water) pair, each order's finish is `max(firstFinish, secondOpen) + secondDuration` тАФ the `max` models waiting for the second ride to open. With `n, m тЙд 100`, brute-force all pairs ├Ч both orders in O(n┬╖m). Both orders must be tried because waiting is asymmetric.

### Day 30 тАФ Earliest Finish Time for Land and Water Rides II
**The trick:** Same problem, but `n, m тЙд 5┬╖10тБ┤` kills the O(n┬╖m) brute force. The two orders **decouple**: since `max(landFinish, waterOpen) + waterDur` is monotonic in `landFinish`, the best land ride (earliest completion) is optimal against *every* water ride. Precompute each category's best completion time, sweep the other once тЖТ O(n + m).

### Day 31 тАФ Total Waviness of Numbers in Range I
**The trick:** With `num2 тЙд 10тБ╡`, just brute-force every number and count interior digits that are strict peaks/valleys (loop indices `1 .. len-2`). Strictness matters тАФ `122` and `100` score 0 because of tied neighbours. Numbers with `< 3` digits fall out for free (empty interior loop). O(R┬╖D).

### Day 32 тАФ Total Waviness of Numbers in Range II
**The trick:** `num2 тЙд 10┬╣тБ╡` forces **digit DP** with `f(num2) тИТ f(num1 тИТ 1)`. Carry the previous two digits so placing a new digit completes a triple and reveals whether the middle is a peak/valley. The crux is the **distribution line `wav += w + add * c`** тАФ a peak discovered high in the tree adds +1 to *every* number flowing through that branch. Returning `(count, sum)` together is what makes summing-over-a-range work.

### Day 33 тАФ Left and Right Sum Differences
**The trick:** No need to build both prefix and suffix arrays. The array partitions into `left + nums[i] + right = total`, so `right = total тИТ left тИТ nums[i]` falls out arithmetically. Carry a running `left` and derive `right` each step. O(n) time, **O(1) extra space**.

### Day 34 тАФ Create Binary Tree From Descriptions
**The trick:** Index nodes by their unique value in a `dict`, creating each node lazily via a `get()` helper so the same physical node is reused wherever it appears. The root is the one value **never seen as a child**. тЪая╕П Gotcha: don't redefine `TreeNode` on LeetCode тАФ it shadows the judge's class and the serializer rejects your return with a `TypeError`.

### Day 35 тАФ Partition Array According to Given Pivot
**The trick:** It's a **stable** three-way partition тАФ relative order within the "less" and "greater" groups must survive, which rules out the swap-based Dutch National Flag. Bucket elements into `less` and `greater` in scan order (preserving order for free), count the pivots, then stitch `less + [pivot]*count + greater`. O(n).

### Day 36 тАФ Maximum Total Subarray Value I
**The trick:** For any subarray, `max(sub) тЙд globalMax` and `min(sub) тЙе globalMin`, so its value `тЙд globalMax тИТ globalMin` тАФ and the whole array hits that bound exactly. Since repeats are allowed, just pick the whole array `k` times: the answer collapses to **`k ├Ч (globalMax тИТ globalMin)`**. (Watch 64-bit overflow тАФ up to ~10┬╣тБ┤.)
